# Changes Summary — `better-docker` Branch

This document describes all changes introduced in the `better-docker` branch relative to `main`, including both original work and changes merged from the upstream (`source`) repository.

---

## 1. Docker Build Modernization

### Multi-stage Dockerfile with `uv` (`Dockerfile.simple`)

The Dockerfile was rewritten from a single-stage `pip install` approach to a two-stage build using [uv](https://github.com/astral-sh/uv), a fast Python package manager.

**Stage 1 — Builder:**
- Uses `uv` (pinned to `0.6.6`) to install dependencies into an isolated virtual environment.
- Reads from `pyproject.toml` + `uv.lock` for fully reproducible, locked dependency resolution.
- No dev dependencies are installed (`--no-dev`).

**Stage 2 — Runtime:**
- Copies only the pre-built `.venv` from the builder stage — no build tools, no pip, no cache in the final image.
- Installs minimal OS packages: `ca-certificates`, `gosu`, `procps`.
- Creates a non-root `appuser` with `/sbin/nologin` shell (hardened vs. the previous `/bin/bash`).
- Tightens directory permissions: `chmod 770` on `/app/logs` and `/app/tmp` (was `777`).
- Uses exec-form `HEALTHCHECK CMD` instead of shell-form for proper signal handling.
- Adds OCI-standard image labels including `org.opencontainers.image.source`.

**Why this matters:** Smaller image, faster builds, deterministic installs, no pip/setuptools attack surface in production.

### Entrypoint Script (`src/docker-entrypoint.sh`)

New entrypoint script that runs as root on container start to:
1. Append a custom CA certificate to the system trust store if `CUSTOM_CA_CERT_PATH` is set — this makes Python's `requests`, `httpx`, and `aiohttp` trust it automatically without per-library configuration.
2. Drop privileges to `appuser` via `gosu` before executing the main process.

### `.dockerignore`

New deny-all allowlist approach: blocks everything by default, then explicitly allows only `pyproject.toml`, `uv.lock`, and `src/**`. This prevents secrets, docs, and build artifacts from leaking into the Docker build context.

## 2. Package Management Migration

### `pyproject.toml` (new)

Added a standard `pyproject.toml` as the single source of truth for project metadata and dependencies. Mirrors the dependencies from `requirements.txt` with the same version constraints. Includes a `[dependency-groups] dev` section for test/lint tools.

### `requirements.txt`

Updated `fastmcp` from `>=2.14.0` to `~=3.1.0` (major version bump). This aligns with the fastmcp v3 API which changes how `stateless_http` is configured and how tool handlers are registered.

### `uv.lock` (new)

Lock file generated by `uv` — pins every transitive dependency to an exact version for reproducible builds. This is the file that `uv sync --frozen` reads during Docker builds.

## 3. MCP Transport Mode — STDIO Support

### Configurable Transport (`enterprise_main.py`)

The server now supports both `streamable-http` (default) and `stdio` transport modes, controlled by the `MCP_TRANSPORT` environment variable.

- `streamable-http`: The existing HTTP-based transport for Q Developer and multi-tenant deployments.
- `stdio`: For MCP client-managed lifecycle (e.g., local development, direct MCP client integration).

**Logging is transport-aware:** HTTP-specific log messages (host/port, health check URL, OAuth callback URL) are only emitted when running in HTTP mode. STDIO mode gets a single clean startup message.

### `stateless_http` Configuration

Two complementary mechanisms ensure stateless HTTP mode is active:
- `os.environ["FASTMCP_STATELESS_HTTP"] = "true"` — env var approach (from upstream, required by newer fastmcp versions).
- `stateless_http=True` passed to `self.mcp.run(...)` — explicit runtime parameter.

### Default Transport Alignment

`run()` method default changed from `"sse"` to `"streamable-http"` to match `run_sync()` and avoid confusion if called without arguments.

## 4. Environment Credential Loading

### SAP Credentials from Environment Variables (`enterprise_main.py`)

When `CREDENTIAL_PROVIDER=env` (the default), the server now reads `SAP_HOST`, `SAP_CLIENT`, `SAP_USERNAME`, `SAP_PASSWORD`, `SAP_LANGUAGE`, and `SAP_INSTANCE_NUMBER` from environment variables and loads them into the in-memory keychain.

This bridges the gap between the simple env-var configuration that users expect and the enterprise tool path (`_get_sap_client_keychain`) which looks up credentials by system identifier.

- System identifier defaults to `env-{SAP_CLIENT}` or can be overridden with `DEFAULT_SAP_SYSTEM_ID`.
- Password is read via `SecretReader.get_secret_or_env()` which checks Docker secrets first, then falls back to the environment variable.

### Public Keychain Storage Method (`keychain_manager.py`)

Added `store_credentials_by_identifier(identifier, credential_data)` — a public method for storing pre-assembled credential dictionaries under a specific lookup key. This replaces direct access to the private `_memory_store` attribute, making the code resilient to internal refactors of the keychain manager.

## 5. Upstream Merge (from `source/main`)

The following changes were merged from the upstream repository:

### Async Tool Handlers (`fastmcp_server.py`)

All MCP tool handler functions were converted from synchronous functions using `asyncio.create_task()` to proper `async def` with `await`. This is the correct pattern for fastmcp v3 and eliminates the task-wrapping indirection.

### SAP Client Improvements (`sap_client.py`)

Minor improvements to the SAP ADT client, including better error handling and connection management.

### Cleanup

- Deleted `assets/` directory (LFS-tracked tar file no longer needed).
- Deleted `deployment/` directory.
- Deleted `.gitattributes` (LFS tracking rules removed along with the tracked files).
- Updated `main.py` to use `streamable-http` transport.

## 6. Build Tooling

### `Makefile` (new)

Convenience targets for Docker and dev workflows:
- `make build` — Build the Docker image for `linux/amd64`.
- `make run` — Run an interactive shell in the container.
- `make docker-export` — Build and export the image as a compressed tarball.
- `make clean` — Remove the tarball and Docker image.
- `make lint` — Lint all Python source with ruff.
- `make lint-changed` — Lint only files changed vs `main` branch.
- `make lint-fix` — Auto-fix lint issues across all source.
- `make lint-fix-changed` — Auto-fix only changed files.

### Ruff Linter (`pyproject.toml`)

Added `ruff` as a dev dependency, replacing `black` and `flake8`. Configuration in `pyproject.toml`:
- Target: Python 3.12, line length 120.
- Rule sets: `E`, `F`, `I`, `W`, `UP`, `B`, `SIM`, `RUF` (with `E501` ignored — handled by formatter).
- isort configured with `aws_abap_accelerator` as first-party.

### `.gitignore`

Added `*.tar.gz` to prevent exported Docker images from being accidentally committed.

## 7. Code Formatting

The `enterprise_main.py` file was reformatted to be consistent with Black code style:
- Single quotes → double quotes throughout.
- Long function signatures and log messages wrapped to respect line length.
- Trailing whitespace and inconsistent blank lines cleaned up.
- Trailing commas added to multi-line data structures and function calls.

These are cosmetic changes with no behavioral impact.

## 8. Kiro Steering Files (`.kiro/steering/`)

Added three steering files to provide Kiro with persistent project context:

- `product.md` — Product overview: what the ABAP Accelerator does, its two deployment modes (standard vs enterprise), supported SAP tools, and target environments.
- `structure.md` — Full project directory layout with descriptions of every module, plus key architectural patterns (entry points, MCP tool registration, SAP client layer, auth subsystem, etc.).
- `tech.md` — Tech stack reference: Python 3.12, uv, key dependencies with versions, ruff linter config, common `make` commands, and Docker build details.

These files are automatically included in Kiro conversations so the AI has accurate context about the codebase without needing to re-discover it each time.
